# LLM Analytics Product

Мини-продукт для аналитики данных с LLM: веб-интерфейс в формате чата, куда можно загрузить `CSV/Excel` или отправить текстовый запрос.  
Модель отвечает текстом, а графики строятся прямо в UI через `Plotly`.

**LLM, используемая в работе (текущая в `.env`):** `MiniMax M2.7` (`minimax/minimax-m2.7`)

## Что делает приложение

- принимает запрос в чат;
- поддерживает загрузку `csv/xlsx/xls` через кнопку-скрепку;
- может распознать таблицу, вставленную текстом;
- отправляет контекст в LLM через API;
- показывает текстовый ответ потоком (streaming);
- строит графики в UI по JSON-спецификации от модели;
- хранит историю чатов отдельно по пользователю в кэше.

## Как работает пайплайн

1. Пользователь отправляет текст и/или файл.
2. Таблица читается в `pandas`, валидируется размер и формат.
3. Формируется контекст датасета (колонки, пропуски, сэмпл строк, summary числовых полей).
4. Идёт запрос в LLM на текстовый ответ.
5. Отдельно (при запросе на графики) идёт запрос в LLM на JSON-спеки графиков.
6. Спеки нормализуются и фильтруются по запросу пользователя.
7. `Plotly` отрисовывает графики в чате.

## Технические особенности

### 1) Потоковый ответ (streaming)

- Реализован в `llm_stream.py` (`SSE`-чтение `data:` чанков).
- Во время генерации в чате показывается «живой» набор текста.
- Если stream не поддерживается или пришёл пустой поток:
  - есть автоматический fallback на обычный API-запрос;
  - есть дополнительные авто-попытки (без ручного повторного ввода).

### 2) Промпты и роль модели

- В коде задан системный промпт, который фиксирует роль LLM как аналитического ассистента.
- Модель отвечает как аналитик **на любой запрос** (в том числе общий), через факты, метрики и выводы.
- Для текстового ответа есть отдельные правила: меньше «воды», больше конкретики по вопросу.
- Для графиков используется отдельный промпт: модель возвращает JSON-спецификацию, а не Python-код.

### 3) История и кэш (24 часа)

- История хранится в памяти процесса через `st.cache_resource`.
- TTL кэша: `24 * 60 * 60` секунд.
- Данные хранятся отдельно по `user_id`.
- `user_id` берётся из query-параметра `uid` (если его нет, создаётся автоматически).
- База данных не используется.

### 4) Графики в UI

- Модель возвращает **только JSON-спеки**, а не исполняемый Python-код.
- Отрисовка выполняется локально через `plotly.express`.
- Поддерживаемые типы: `bar`, `line`, `scatter`, `pie`, `histogram`.
- Есть нормализация:
  - авто-подстановка метрики в `y`, если модель вернула «пустой» spec;
  - защита от «бесполезных» графиков `count`, когда нужна реальная метрика.

## Что где находится

- `app.py`
  - Streamlit UI;
  - чат, загрузка файлов, история диалогов;
  - связывает текстовый поток и построение графиков.
- `llm_client.py`
  - загрузка конфига из `.env`;
  - запросы к LLM API;
  - генерация текста и JSON-спеков графиков.
- `llm_stream.py`
  - потоковый режим (SSE) для ответа модели.
- `analytics_core.py`
  - нормализация спеков графиков;
  - fallback-логика по графикам;
  - фильтрация по типу/количеству графиков;
  - сборка `plotly`-фигур.
- `chat_cache.py`
  - хранение истории и очистка просроченного кэша.

## Переменные окружения (`.env`)

Пример в `.env.example`.

```env
LLM_PROVIDER_LABEL=Example Provider
LLM_API_BASE_URL=https://api.example.com/v1
LLM_CHAT_COMPLETIONS_PATH=/chat/completions
LLM_MODEL=provider/model-name
LLM_API_KEY=sk-example-key

LLM_TIMEOUT_SECONDS=90
LLM_TEMPERATURE=0.2
LLM_MAX_TOKENS=1200
LLM_MAX_CONTEXT_ROWS=30
LLM_MAX_CHARTS=3
```

### Комментарии по настройке

- `LLM_MAX_TOKENS` — длина ответа модели.
- `LLM_MAX_CONTEXT_ROWS` — сколько строк датасета отправлять в контекст.
- `LLM_MAX_CHARTS` — верхний лимит графиков за ответ (дополнительно ограничивается текстом запроса).

## Запуск проекта

### 1) Локально (venv)

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
cp .env.example .env
```

Заполните `.env` и запустите:

```bash
streamlit run app.py
```

Открыть:

```text
http://localhost:8501
```

### 2) Через Docker (локально)

```bash
docker compose up --build
```

Открыть:

```text
http://localhost:8501
```

## Частые проблемы

1. `В .env не задано обязательное поле ...`
   Проверьте заполнение `.env`.

2. `Модель не вернула текстовый ответ` / пустой ответ
   В проекте есть авто-fallback и авто-retry, но при нестабильном API провайдера могут быть деградации.

3. История пропала
   Кэш хранится в памяти процесса. При перезапуске контейнера/процесса история сбрасывается.
